## Context

Governance APIs will primarily revolve around sensitive content policies and reports. These are configurations you could create and set under the Admin -> Governance section of Glean. Once set, they will scan all the documents indexed by Glean based on the configuration provided in the policies/reports. Finally all the violations or sensitive content will be found and will be available to review.

Document hiding and unhiding is also possible through the **Sensitive Findings** and **Content Hiding** dashboards. We also provide a couple of APIs to cover these actions.


## Difference between policies and reports.

Policies are the newer and more refined version of reports. Reports used to be the older way to do sensitive content scans. They would provide the option to download all the violations as a CSV once the scan completes. 

Policies provide an enhanced UI dashboard to view all the findings and take certain actions on them. You will have the ability to archive findings, hide documents which have sensitive information etc.

Ideally policies would completely replace reports. However, policies have only 2 frequencies supported: WEEKLY and CONTINUOUS.

Reports also support one-time runs where a sensitive content scan is done immediately and never again. In the APIs explained below, we will support only this feature of Reports as it is the only thing it has that policies do not.

NOTE: The policy and report APIs use many common internal structures so there could be some redundancy in fields. Make sure to keep them consistent throughout the payload.


# Authorization Token

In order to use the policy and report endpoints, you would need to create a Client API Token with the DATA\_GOVERNANCE scope. While creating the token, the user email-id embedded in the token must be of someone in the org who has access to the Sensitive Content Pages under admin section. This could be a Super Admin or a Sensitive Content Moderator.

Once the token is created, it can be passed in the header of the requests to authenticate them.

For the content hiding APIs, you would need to do the same as above but use the CONTENT\_HIDING scope. The user has to be one with access to editing document exclusions which could be a super-admin.

The process of creating these tokens cane be found here : [Managing API Tokens](https://developers.glean.com/docs/indexing_api_tokens/).


# The APIs

The APIs will be broadly divided into 3 sections. Policy, Report and Content Hiding endpoints. 

Both Reports and Policies have a common "config" with the following important fields.

**Config -**

| Field Name | Description |
|-----|-----|
| `Version` | This is a unique identifier for the config. It will be synonymous with the report or policy id. It is NOT the same as the policy-version. |
| `InputOptions` | This will control all the data sources and time range of documents to be considered for the scan.<br/>• _datasourcesType_ can be set to ALL to consider all datasources. Or it can be set to CUSTOM in which case, the specific _datasourceInstances_ need to be populated.<br/>• _timePeriodType_ can be set to ALL_TIME, PAST_YEAR, or CUSTOM. If it is set to CUSTOM, the _customTimeRange_ field must be set with the start and end time following ISO date-time format.<br/><br/>Note: The last edited time of the document would be considered for this filter. |
| `BroadSharingOptions` | This controls how widely shared the document must be to be picked up by the scan.<br/>• _anyoneInternalEnabled_ will pick up documents that are accessible to the entire org.<br/>• _anonymousAccessEnabled_ will pick up documents accessible to the entire internet.<br/>• _thresholdEnabled_ will allow filtering based on the number of people who have access to a document and the _threshold_ field would have to be set to control the same.<br/>• _userAccessEnabled_ will filter in documents which are accessible to any of the users present in _userIds_. |
| `sensitiveContentOptions` | This is when you will actually define the sensitive content you would like to scan for. It can be specified in 3 ways:<br/>• _sensitiveInfoTypes_ are predefined categories that we will use Google's DLP API to determine if the content in a doc falls into these categories or not.<br/>• _sensitiveTerms_ are plain text terms that would flag any documents with the term.<br/>• _sensitiveRegexes_ are regular expressions that must follow the RE2 syntax which will flag any doc with content matching one of the regexes. |
| `reportName` | Synonymous with policy or report name. |
| `Frequency`                  | Interval between scans. |
| `CreatedBy`                  | Person who created the policy/report. |
| `CreatedAt`                  | Timestamp when the config was created. |
| `redatedQuote`               | Whether the sensitive quote should be redacted in the final violation report. |
| `autoHideDocs`               | Whether documents must be auto-hidden if flagged. |
| `allowlistOptions`           | List of terms that will NOT flag a document even if it is picked up by a sensitive content option. |

## Policy APIs

With these endpoints, you can perform CRUD operations on policies. This will be synonymous to using the policy dashboard in the Sensitive Findings section under the Admin > Governance section in the Glean UI.

**Important fields in a policy:**

| Field Name| Description |
|---------|------------|
| `Name` | This would be the name of the policy which would be displayed in the UI. |
| `Id` | Unique numeric identifier for a policy. |
| `Version` | This is the version of the policy which keeps incrementing every time the policy is edited. |
| `Frequency` | This is the frequency at which the policy will run. Same as the config frequency. |
| `AutoHideDocs` | This would control whether any documents flagged by this policy would get auto-hidden. |
| `Status` | This is the active state of the policy. It will run scans if it is "ACTIVE" based on its frequency. "INACTIVE" implies it will not execute scans but shows as "archived" in UI. "CANCELLED" policies will not show up. |
| `createdBy` | Same as config _createdBy_. |
| `createdAt` | Same as config _createdAt_. |
| `lastUpdatedAt` | Timestamp at which the policy was last updated. |
| `Config` | See common config for policies and reports. |


### Creating a policy.

This will create a new policy with all the specified configurations. The payload will need the name of the new policy, its configuration, frequency and autoHideDocs. 

In the config, the **createdAt** and **version** field need not be set. **CreatedBy** is optional.

Once created, the entire policy will be returned back along with its unique **id**.

Endpoint : POST https://{domain}.glean.com/rest/api/v1/governance/data/policies

Example :
```bash
curl -X POST 'https://{domain}.glean.com/rest/api/v1/governance/data/policies' \
-H 'Authorization: Bearer ABCTOKEN=' \
-H 'Content-Type: application/json' \
--data-raw '{"autoHideDocs":false,"config":{"inputOptions":{"datasourcesType":"CUSTOM","timePeriodType":"PAST_YEAR","datasourceInstances":["gdrive"]},"broadSharingOptions":{"threshold":5,"thresholdEnabled":true,"anyoneInternalEnabled":false,"anonymousAccessEnabled":false,"userAccessEnabled":false,"userIds":[]},"sensitiveContentOptions":{"sensitiveInfoTypes":[{"infoType":"PASSWORD"}],"sensitiveTerms":[{"expression":"oxygen"}],"sensitiveRegexes":[{"expression":"sampleregex"}]},"reportName":"example-name","frequency":"CONTINUOUS","redactQuote":false,"autoHideDocs":false,"allowlistOptions":{"terms":["allowed"]}},"frequency":"CONTINUOUS","name":"example-name"}'
```

### Updating a policy.

This will update an existing policy. Specifically the policy with the same is as the one passed in the path. Any one of the fields _config, frequency, status, autoHideDocs, name_ must be passed for the update call to pass. The unset fields will remain the same. Whether the update was successful or not would be returned.

In the config, the **createdAt** field need not be set. **CreatedBy** is optional.

Endpoint : POST https://{domain}.glean.com/rest/api/v1/governance/data/policies/{id}

Example :
```bash
# Update all fields at once.
curl -X POST 'https://{domain}.glean.com/rest/api/v1/governance/data/policies/1234567890' \
-H 'Authorization: Bearer ABCTOKEN=' \
-H 'Content-Type: application/json' \
--data-raw '{"autoHideDocs":false,"config":{"inputOptions":{"datasourcesType":"ALL","timePeriodType":"CUSTOM","customTimeRange":{"startTime": "2020-03-08T18:30:00Z","endTime": "2024-08-23T18:29:59Z"}},"broadSharingOptions":{"threshold":5,"thresholdEnabled":true,"anyoneInternalEnabled":false,"anonymousAccessEnabled":false,"userAccessEnabled":false,"userIds":[]},"sensitiveContentOptions":{"sensitiveInfoTypes":[{"infoType":"PASSWORD"}],"sensitiveTerms":[{"expression":"oxygen"}],"sensitiveRegexes":[{"expression":"sampleregex"}]},"reportName":"example-name","frequency":"CONTINUOUS","redactQuote":false,"autoHideDocs":false,"allowlistOptions":{"terms":["allowed"]}},"frequency":"CONTINUOUS","repotName":"example-name", "status":"INACTIVE"}'

# Update only one field (others will remain same as before).
curl -X POST 'https://{domain}.glean.com/rest/api/v1/governance/data/policies/1234567890' \
-H 'Authorization: Bearer ABCTOKEN=' \
-H 'Content-Type: application/json' \
--data-raw '{"status":"ACTIVE"}'
```

### Getting a policy.

This will fetch an existing policy matching the id provided. The policy version can also be optionally provided to get an older version of the policy if it has been edited.

The entire policy will be returned.

Endpoint : GET https://{domain}.glean.com/rest/api/v1/governance/data/policies/{id}

Example :
```bash
# Not specifying version will get the latest policy.
curl -X GET 'https://{domain}.glean.com/rest/api/v1/governance/data/policies/1234567890' \
-H 'Authorization: Bearer ABCTOKEN='

# Specifying version will get a specific edit version of the policy.
curl -X GET 'https://{domain}.glean.com/rest/api/v1/governance/data/policies/1234567890?version=0' \
-H 'Authorization: Bearer ABCTOKEN='
```

### Listing multiple policies.

This will fetch all the existing policies with some filtering. You can provide the **autoHide** and **frequency** parameters to filter on.

Endpoint : GET https://{domain}.glean.com/rest/api/v1/governance/data/policies

Example :
```bash
curl -X GET 'https://{domain}.glean.com/rest/api/v1/governance/data/policies?frequency=WEEKLY&autoHide=true' \
-H 'Authorization: Bearer ABCTOKEN='
```

### Downloading policy violations.

This will download a CSV file of all the violations of a specific policy's latest scan. 

Note: this is only supported for weekly policies. Continuous policies will have their findings continuously dumped into your cloud storage from where you can consume them directly.

Endpoint : GET https://{domain}.glean.com/rest/api/v1/governance/data/policies/{id}/download

Example :
```bash
curl -X GET 'https://{domain}.glean.com/rest/api/v1/governance/data/policies/1234567890/download' \
-H 'Authorization: Bearer wu21k2dSwONsioXa5MBkYj0FDc0xDtEuMbeGbIdkkGI=' > output.csv
```

## Report APIs

Only creating a one-time report is supported via API for reports. You can get the report's scan status to check if it has completed and download its violation CSV. For any of the other features such as updating, other frequencies, etc., please use policies.

Reports created using this endpoint will appear in the "Sensitive Content Reporting" section under Admin > Governance in the UI.


### Creating and running a One-Time report.

This will create a report and immediately trigger its sensitive content scan. The **frequency** field must be populated with ONCE.

In the config, the **createdAt** and **version** field need not be set. **CreatedBy** is optional.

Endpoint : POST https://{domain}.glean.com/rest/api/v1/governance/data/reports

Example :
```bash
curl -X POST 'https://{domain}.glean.com/rest/api/v1/governance/data/reports' \
-H 'Authorization: Bearer ABCTOKEN=' \
-H 'Content-Type: application/json' \
--data-raw '{"config":{"inputOptions":{"datasourcesType":"ALL","timePeriodType":"ALL_TIME"},"broadSharingOptions":{"enabled":true,"threshold":5,"thresholdEnabled":true,"anyoneWithLinkEnabled":true,"anyoneInternalEnabled":true,"anonymousAccessEnabled":true,"userAccessEnabled":false,"userIds":[]},"sensitiveContentOptions":{"sensitiveInfoTypes":[],"sensitiveTerms":[{"expression":"oxygen"}],"sensitiveRegexes":[]},"reportName":"example-name","redactQuote":false},"frequency":"ONCE"}'
```

### Getting Report Scan Status.

This will get the current status of the Report's scan.

| `Status` | Description |
|----------|-------------|
| `PENDING` | Implies that the run is in progress. |
| `SUCCESS` | Implies the run is complete and the violation can be downloaded. |
| `FAILURE` | Implies there was some problem during the execution of the scan and it has failed. You would have to retry in this case. |

It also returns the timestamp at which the scan started.

Endpoint : GET https://{domain}.glean.com/rest/api/v1/governance/data/reports/{id}/status

Example :
```bash
curl -X GET 'https://{domain}.glean.com/rest/api/v1/governance/data/reports/1234567890/status' \
-H 'Authorization: Bearer ABCTOKEN='
```

### Downloading Report violations.

This will download a CSV file of all the violations of a specific report.

Endpoint : GET https://{domain}.glean.com/rest/api/v1/governance/data/reports/{id}/download

Example :
```bash
curl -X GET 'https://{domain}.glean.com/rest/api/v1/governance/data/reports/1234567890/download' \
-H 'Authorization: Bearer ABCTOKEN='
```

## Content Hiding Apis

Content hiding works by setting a "visibility override" to a particular document. These overrides can come from different sources, such as: 

1. Existing policies that might flag a document meeting its requirements if "auto-hide" is enabled.

2. If it is manually hidden by a user. Different users are treated as different sources.

3. If a document is present in the CSV for hiding it.

Each of these overrides is distinct and if any one of them hide a document, it will be hidden. This rest endpoint only gives you control over the "manual override by user".


### Visibility overrides

Visibility overrides refer to the visiblity of a document that is being enforced from a particular source. For example, policy A might want the document to be visible, while policy B might want it to be hidden. All these sources are considered together and the source with the highest visiblity override is considered. A user can also be a source of visibility override which is what the below API provides.

**Important fields in a Visibility overrides:**

| Field Name| Description |
|---------|------------|
| `docId` | The id of the document whose visiblity override is considered. |
| `override` | The visibility override status of the document. |


There are three types of visibility overrides status.

| `Override Status` | Description |
|----------|-------------|
| `HIDE_FROM_ALL` | The document is hidden from all Glean users. |
| `HIDE_FROM_ALL_EXCEPT_OWNER` | The document is hidden from all Glean users except for the owner. If there is no owner, it remains hidden from everyone. |
| `HIDE_FROM_GROUPS` | The document is hidden from all users except those explicitly shared with. |


### Submitting visibility overrides for documents

You can submit a list of document ids along with the desired visibility override for each one. This will set the "user" level visibility override for the document. 

The returned value contains the documents along with their visibility override values and whether the action was successful for them. This endpoint could have a partial failure which can be deduced from the **success** field in the return parameter.

Endpoint : POST https://{domain}.glean.com/rest/api/v1/governance/documents/visibilityoverrides

Example :
```bash
curl -X POST 'https://{domain}.glean.com/rest/api/v1/governance/documents/visibilityoverrides' \
-H 'Authorization: Bearer ABCTOKEN=' \
--data-raw '{"visibilityOverrides": [{"docId":"doc1","override":"NONE"}, {"docId":"doc2","override":"HIDE_FROM_GROUPS"}]}'
```

### Getting visibility overrides for documents

You can submit a list of document ids and get the current final visibility override for the documents passed.

Endpoint : GET https://{domain}.glean.com/rest/api/v1/governance/documents/visibilityoverrides

Example :
```bash
curl -X GET 'https://{domain}.glean.com:7962/rest/api/v1/governance/documents/visibilityoverrides?docIds=doc1&docIds=doc2&docIds=doc3' \
-H 'Authorization: Bearer ABCTOKEN='
```
